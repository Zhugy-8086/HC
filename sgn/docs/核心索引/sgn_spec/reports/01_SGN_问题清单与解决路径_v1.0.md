## 一、战略层面问题（7个核心断层）

### 问题 1: 竞争稳态未证 —— 缺纯整数竞争学习的端到端稳态报告

**本质**  
SGN 是离散整数竞争网络，无反向传播、无 loss 曲面。其"训练"是模板演化（追加→合并→淘汰），收敛标准不是 loss 曲线，而是**模板库规模稳态**与**识别率稳态**。当前 38 份规范全是"可以怎么做"，没有一份给出"从第 0 步到第 N 步，模板库规模从 0 收敛到稳态 80±5 个"的实证。

**给现有整数社区的转接意义**  
PyTorch Quantization / TFLite 全整数推理社区需要看到：纯整数体系下，竞争学习不会发散。这是吸引社区迁移的**门票**。

**怎么做（解决路径）**  
1. **定义稳态指标**: 连续 500 步内模板库规模波动 < 10%，识别率波动 < 2%。
2. **Python 验证器**: 在 `sgn_core.py` 中增加稳态日志，每步输出 `(step, template_count, recognition_rate, top5_accuracy)`。
3. **收敛报告**: 在标准数据集（如 MNIST 4×4 二值化）上运行 5000 步，绘制 `template_count vs step` 曲线，证明单调收敛到稳态。
4. **C 移植验证**: 将同一组随机种子在 PC Python 端和 AI8051U C 端各跑一遍，对比稳态时的模板库 md5sum。
5. **文档产出**: 新增《SGN 竞争学习稳态收敛报告 v1.0》，作为 L0 核心文档。

### 问题 2: 热同步缺失 —— 缺 PC-MCU 增量模型同步协议

**本质**  
38 份规范定义了存储格式（13 字节固定记录）、UART 帧格式、Merkle 树完整性校验，但**缺少从 PC 训练端到 MCU 推理端的增量同步语义**。现有链路是：
```
PyTorch 训练 → ONNX 导出 → TFLite 转换 → MCU 烧录
```
SGN 需要替代这条链路，但文档里没有：
- 增量 diff 协议：Merkle 树可以做子树 diff，但哪种子树变化需要同步？
- OTA/热更新帧格式：UART 规范有基础帧，但没有"模型增量同步帧"（TYPE=0x04 仅预留，未定义字段）。
- 版本兼容性：MCU 端 level=2，PC 端 level=6，运行时高精度梯度如何映射到低精度权重？

**怎么做**  
1. **增量 diff 协议**: 基于 Merkle 树的子树哈希比对，定位变更模板。定义 `MERKLE_DIFF` 帧：
   ```
   [0xAA][0x55][TYPE=0x21][LEN][SUBTREE_ROOT:2][OLD_HASH:8][NEW_HASH:8][PAYLOAD:N][CRC8]
   ```
2. **热更新帧**: 定义 `TYPE=0x04` 为模型同步帧，子类型：`0x01`=全量推送, `0x02`=增量推送, `0x03`=回滚请求。
3. **精度映射**: 定义 `level_downgrade()` 函数，PC 端 level=6 模型加载到 MCU level=2 时，前 4 层保留，后 2 层截断，并给出截断误差上界（< 2.33e-10）。
4. **版本协商**: UART 握手阶段交换 `SGN_HPDC_LEVEL` 和 `MODEL_VERSION`（uint32），不一致时触发自动转换或拒绝加载。
5. **文档产出**: 新增《SGN_PC-MCU 增量同步协议规范 v1.0》，归入 L2 桥接层。

### 问题 3: 社区接口封闭 —— 缺 ONNX/PyTorch/MLIR 桥接规范

**本质**  
38 份规范是**自说自话的封闭宇宙**。整数网络社区（PyTorch Quantization, TFLite, ONNX Runtime QDQ, Apache TVM/VTA, IBM Neuroblast）看到 38 份规范，只会觉得"很厉害，但怎么接入我的 PyTorch pipeline？"

**怎么做**  
1. **ONNX → HC 映射规范**: 定义 ONNX 算子到 SGN 算子的映射表：
   - `QuantizeLinear` → `HC8.from_float()`
   - `Conv` → `Trie索引 + WTA竞争`（需说明 SGN 无卷积，这是概念映射）
   - `MaxPool` → `WTA竞争算子`
   - `ReLU` → `软阈值稀疏诱导`
2. **PyTorch 训练钩子**: 提供 `sgn_pytorch_bridge.py`，允许用户在 PyTorch 训练循环中插入 SGN 模板库作为"竞争后处理层"。
3. **MLIR Dialect 定义**: 定义 `sgn` MLIR dialect，包含 `sgn.trie`, `sgn.wta`, `sgn.soft_threshold` 等操作，让 TVM/VTA 后端可以识别并优化。
4. **示例代码**: 提供 "PyTorch QAT 模型 → SGN 模板库" 的端到端迁移示例（如 ResNet-8 二值化后导入 SGN）。
5. **文档产出**: 新增 `sgn_extensions/pytorch_bridge.md`、`sgn_extensions/onnx_mapping.md`、`sgn_extensions/mlir_dialect.md`，归入 L2 桥接层。

### 问题 4: 浮点依赖未闭环 —— 沙盒层 float64/float128 的脱离条件未给出

**本质**  
投影除法沙盒规范明确承认："HC 负责存和传，float64/float128 负责算，算完立即回灌。" 这暴露了深层矛盾：SGN 声称"真整数"，但 PC 端复杂运算仍依赖浮点。

**怎么做**  
必须证明两件事之一：
- **路径 A**: 沙盒层的浮点运算可以被纯整数替代，且速度/精度不劣化。
  - 除法：Hensel 求逆元（奇数除数）+ 长除法（偶数除数）已覆盖，需给出与 float64 除法的精度对比表。
  - 开方：2-adic Hensel 提升已覆盖，需给出收敛步数与精度关系表（45 步填满 48 位）。
  - 梯度更新：SGN 无梯度，此条不适用。但沙盒内的 "gradient_update" 函数需说明仅用于 PC 端兼容实验，非 SGN 核心。
- **路径 B**: 沙盒层的浮点是临时翻译层，最终会收敛到一套固定的整数参数，之后完全脱离浮点。
  - 定义 "沙盒收敛条件"：当投影回灌后的 HC 值连续 100 次迭代变化 < 1 LSB（最低有效位）时，视为收敛，冻结为纯整数参数。
  - 提供 `sandbox_freeze()` 接口，将浮点沙盒输出固化为仅 HC 模式的常量表。

**文档产出**: 在《投影除法沙盒规范》中新增第 10 章 "脱离浮点的收敛条件与冻结接口"。

### 问题 6: 内存预算未闭合 —— 34KB SRAM 下多规范共存无全局统筹

**本质**  
各规范独立声称内存占用，但没有一份**全局内存预算表**说明在 34KB SRAM 下哪些可以共存、哪些必须互斥。

**验算结果（关键修正）**  
- AI8051U SRAM: 34KB = 34,816 字节
- 系统开销（栈+中断+全局）: ~4KB
- **实际可用**: ~30,720 字节

| 配置 | 占用 | 可行性 |
|------|------|--------|
| 最小核心 (WTA+Trie+256N) | 8,304 B | ✅ 余量 22KB |
| 标准核心 (+Merkle+CBF256) | 10,952 B | ✅ 余量 19KB |
| 高可靠 (+TMR部分+RS) | 12,176 B | ✅ 余量 18KB |
| **全量豪华** (TMR全量+CBF1024+Merkle256+Fréchet+DFA) | **30,784 B** | ❌ **超支 64 B** |

**怎么做**  
1. **制定内存预算表**: 在 `sgn_config.h` 中增加内存分配宏：
   ```cpp
   #define SGN_SRAM_TOTAL     34816
   #define SGN_SRAM_RESERVED  4096   // 系统栈+中断
   #define SGN_SRAM_BUDGET    (SGN_SRAM_TOTAL - SGN_SRAM_RESERVED)
   ```
2. **互斥矩阵**: 定义哪些规范不能同时启用：
   - `TMR_FULL` 与 `CBF_1024` 互斥（两者合计 > 16KB）
   - `FRÉCHET_FREQ_TABLE` 与 `TMR_FULL` 互斥
   - `RS_TABLE_MODE`（64KB 查表）与任何配置互斥（Flash 不够），强制使用 `RS_LFSR_MODE`
3. **分层内存池**: 将 SRAM 划分为：
   - 核心池 (16KB): 神经元 + 模板库 + Trie
   - 扩展池 (8KB): Merkle / CBF / LRU（三选一或动态切换）
   - 可靠池 (4KB): TMR 副本 / RS LFSR / 校验和
   - 缓冲池 (2KB): UART / I/O / 看门狗
4. **动态降级**: 当模板库规模 > 128 时，自动关闭 Fréchet 频率表（节省 3.5KB），回退到在线计算。

### 问题 7: 硬件性能数据缺失 —— 整数路径 vs TFPU 路径的延迟/功耗/面积对比

**本质**  
SGN 主动弃用 TFPU，但社区会问："如果整数路径比 TFPU 浮点慢，我为什么要迁移？" 当前文档没有任何对比数据。

**验算结果（48MHz 1T 修正）**  
之前文档普遍按 24MHz 12T（~500ns/机器周期）估算，严重低估了 1T 架构性能：

| 操作 | 24MHz 12T (旧估算) | 48MHz 1T (新验算) | 差异 |
|------|-------------------|-------------------|------|
| HC8 比较 | 12 周期 / 6 μs | 12 周期 / **0.25 μs** | 24× 快 |
| HC8 加法 | 48 周期 / 24 μs | 48 周期 / **1.0 μs** | 24× 快 |
| WTA Trie 剪枝 | 600 周期 / 300 μs | 600 周期 / **12.5 μs** | 24× 快 |
| 软阈值 | 48 周期 / 24 μs | 48 周期 / **1.0 μs** | 24× 快 |
| Merkle 增量更新 | 400 周期 / 200 μs | 400 周期 / **8.3 μs** | 24× 快 |
| TMR 投票 | 50 周期 / 25 μs | 50 周期 / **1.04 μs** | 24× 快 |

**怎么做**  
1. **基准测试**: 在 AI8051U 上实测以下对比：
   - 整数 WTA (Trie+HC比较) vs TFPU 浮点 TopK
   - 整数 HC 加法 vs TFPU float32 加法
   - 整数 Merkle 路径更新 vs 软件 SHA-256
2. **功耗测试**: 使用开发板电流计，对比：
   - 纯整数推理循环（关闭 TFPU 时钟）
   - TFPU 浮点推理循环（开启 TFPU）
   - 混合模式（SGN 核心整数 + 沙盒浮点）
3. **面积说明**: TFPU 是芯片已集成面积，SGN 整数路径不增加额外硅面积；但需说明若用外部浮点库（如软件 float），代码体积 > 8KB，而 SGN 整数核心 < 4KB。
4. **文档产出**: 新增《SGN 整数路径 vs TFPU 性能基准报告》，包含延迟、功耗、代码体积三张对比表。

## 三、技术实现问题（基于验算的新发现）

### 问题 8: CBF 假阳性率数据错误（🔴 高）

**本质**  
《计数布隆过滤器规范 v1.1》声称 `n=256, m=1024, k=3` 时 `P_fp≈1.8%`，但按标准布隆过滤器公式计算：
```
P_fp = (1 - e^(-kn/m))^k = (1 - e^(-3×256/1024))^3 ≈ 14.7%
```
1.8% 与 14.7% 相差 **8 倍**，属于严重数据错误。

**怎么做**  
1. 修正 v1.1 文档中的假阳性率表，以标准公式为基准。
2. 若需达到 1.8% 假阳性率，应给出正确参数：`m=4096, k=3`（此时 P_fp≈1.8%）。
3. 增加说明：HC-CBF 的"梯度阈值"判定（非布尔存在测试）可降低等效假阳性，但设计选型时必须以标准公式为保守上界。

### 问题 9: 投影沙盒 float64 对 HC16/HC32 溢出位数错误（🟡 中）

**本质**  
《投影除法沙盒 v1.1》声称 float64 对 HC16 损失 27 位、对 HC32 损失 75 位。重新验算：
- float64 尾数 53 位
- HC16 总信息量 80 位 → 溢出 `80-53=27` 位 ✅ 正确
- HC32 总信息量 128 位 → 溢出 `128-53=75` 位 ✅ 正确
- **但**: float128 尾数 113 位，对 HC32 溢出 `128-113=15` 位，文档声称"无损"是**错误的**（除非压缩 int_part）。

**怎么做**  
1. 修正 float128 统一沙盒的声明：HC8/HC16 无损，HC32 默认 int_part=32 位时仍有 15 位损失。
2. 明确给出 HC32 的无损条件：通过"位宽压缩"将 int_part 压到 8 位（总信息量 72 位 < 113 位）。

---

### 问题 10: 时序估算过于乐观（🟡 中）

**本质**  
多份规范（如 Hensel 提升、快速幂、排序网络）给出的"人天"估算（如 0.5 天、1 天）基于理想条件，未考虑：
- 8051 的 C 编译器优化水平（Keil AC6 与 GCC 差异）
- 多精度乘法的寄存器压力（48MHz 下 7 字节卷积需要多次访存）
- 测试验证时间（单元测试 + 集成测试通常占 50% 工时）

**怎么做**  
1. 所有"人天"估算乘以 **2.5 倍安全系数**（考虑测试、调试、文档）。
2. 区分"原型实现"（Python 验证器）和"生产固化"（C 头文件 + 8051 实测）的工时。
3. 给出 8051 特有的限制：无 `__builtin_clz`，需用查表法；无 64 位寄存器，多精度运算需拆分为 8 位操作。
