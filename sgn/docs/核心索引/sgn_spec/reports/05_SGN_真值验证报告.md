## 三、逻辑自洽性验证

| 命题 | 验证结果 | 说明 |
|------|---------|------|
| "仅HC模式下线性映射局部可控" ↔ "合体模式下线性映射撕裂球" | ✅ 自洽 | 分离使用文档已明确修正 |
| "时间戳用HC单独，不用合体" ↔ "时间变量处理" | ✅ 自洽 | 两份文档一致 |
| "CBF桶向level进位" ↔ "仅HC模式下CBF level=0" | ⚠️ 矛盾 | CBF规范说"仅HC模式(level=0)，int_part饱和255"，但又称"向level进位，范围无界"。若level=0固定，则无法向level进位，int_part饱和后应停止。需明确：CBF是"仅HC模式"还是"合体模式"？ |
| "投影沙盒仅在PC端" ↔ "MCU端无沙盒" | ✅ 自洽 | 沙盒规范明确标注PC端扩展 |
| "TMR三副本" ↔ "34KB SRAM" | ⚠️ 紧张 | 全量TMR 15KB + 其他组件可能超支，需部分TMR降级 |
| "RS查表法64KB" ↔ "Flash 64KB" | ❌ 矛盾 | 查表法需要64KB Flash，但程序代码至少占20KB，不可能全给RS表 |
| "排序网络N=256" ↔ "SRAM 34KB" | ✅ 自洽 | 排序缓冲2KB，在预算内 |
| "Fréchet频率表7×256×4B=7KB" ↔ "34KB SRAM" | ✅ 自洽 | 但与其他大内存组件共存时需取舍 |
| "霍夫曼前缀码压缩率30-50%" ↔ "模板库幂律分布" | ⚠️ 有条件 | 仅在模板访问频率极度不均（如Zipf分布）时成立。若均匀分布，变长码反而膨胀。文档未给出频率分布实测 |
| "补码负数范围减半(127 vs 255)" ↔ "SGN参数均为正" | ✅ 自洽 | 补码仅用于未来梯度差值场景，当前参数仍用无符号 |
| "形态学膨胀不新增模板" ↔ "激活扩散" | ✅ 自洽 | 膨胀是读取操作，仅标记鼓励 |
| "Lamport时钟无需NTP" ↔ "物理时钟漂移" | ⚠️ 需补充 | 文档承认晶振偏差±1%，但未给出漂移累积的上界分析 |
| "Quorum单轮广播" ↔ "UART半双工" | ⚠️ 需补充 | 半双工总线下"单轮广播"实际是轮询或主从广播，文档未明确总线仲裁机制 |

## 四、硬件可行性验证

| 命题 | 验证结果 | 说明 |
|------|---------|------|
| 所有运算可在8位MCU运行 | ✅ | 纯整数，无64位乘法（多精度拆分为8位） |
| 8051支持`std::array` | ⚠️ | C++11 `<array>` 需要编译器支持。Keil AC6支持，但传统8051 C51编译器可能不支持。需确认目标编译器 |
| 8051支持`uint64_t` | ⚠️ | `<cstdint>` 中的`uint64_t`在8位MCU上通常是软件模拟，运算慢。HC的多精度运算应手动拆分为`uint8_t`数组，而非依赖`uint64_t` |
| 8051支持模板/泛型 | ❌ | `ProjectionSandbox128`的C++模板在8051上不可行。需改为宏或函数指针 |
| 8051支持`__builtin_clz` | ❌ | 8051无CLZ指令。Merkle差异定位中的`__builtin_clz`需改为查表法 |
| 8051支持`long double`(float128) | ❌ | 8051无float128硬件，软件模拟极慢。投影沙盒明确标注"仅PC端"是正确的 |
| 34KB SRAM容纳标准配置 | ✅ | 标准配置~11KB，余量充足 |
| 34KB SRAM容纳全量豪华配置 | ❌ | 超支64B，不可行 |
| 64KB Flash容纳程序+RS查表 | ❌ | RS查表64KB + 程序代码20KB = 84KB > 64KB |
| 64KB Flash容纳程序+RS-LFSR | ✅ | LFSR模式仅需几百字节代码 |
| 48MHz/1T满足实时性 | ✅ | 标准配置每步115μs，远小于典型推理周期50ms |
| UART 9600波特率满足分布式同步 | ⚠️ | 9600下256神经元全量同步需2.1秒，太慢。需115200波特率（178ms）或增量同步 |
| 看门狗1ms tick ISR不阻塞训练 | ✅ | 1.04μs/次，占空比<0.1% |

## 五、严重错误汇总（需立即修正）

| 编号 | 错误 | 来源 | 影响 | 修正方案 |
|------|------|------|------|---------|
| E1 | Hensel提升步数=6步（实际45步） | 2-adic平方根 v1.1 | 误导实现者严重低估开方时间 | 修正为"45步逐位提升，每8步填满1个byte层" |
| E2 | CBF假阳性率1.8%（实际14.7%） | CBF v1.1 | 设计者可能选用过小桶数，导致高假阳性 | 按标准公式重新制表；若需低假阳性，推荐m=4096 |
| E3 | float128对HC32"无损"（实际15位溢出） | 投影除法 v1.1 | 误导用户认为float128万能 | 修正为"HC32默认int_part=32位时仍有15位损失，需压缩到8位才无损" |
| E4 | 性能估算基于24MHz/12T（过时） | 多份文档 | 严重低估48MHz/1T性能，导致"MCU太慢"的虚假结论 | 全部重新按48MHz/1T验算 |
| E5 | RS查表法64KB未说明Flash不可行 | RS v1.0 | 实现者可能在8051上尝试64KB查表，导致Flash爆炸 | 增加醒目警告："查表法仅适用于PC验证器，MCU强制使用LFSR模式" |
| E6 | CBF"向level进位"与"仅HC模式(level=0)"矛盾 | CBF v1.1 | 语义混乱：若level=0固定，则无法进位 | 明确CBF桶的level语义：若仅HC模式，int_part饱和255；若需无界计数，应使用合体模式 |

**总体评价**:  
SGN 技术栈的**数学基础和架构设计是坚实的**，超度量性质、Trie索引、HC运算等核心概念的证明和定义正确。但存在 **5 个严重数据错误**（Hensel步数、CBF假阳性率、float128溢出、性能估算过时、RS查表可行性）和 **3 个硬件可行性矛盾**（全量TMR+CBF1024超SRAM、RS查表超Flash、C++模板不兼容8051）。这些问题需要在文档层面立即修正，并通过实测验证关键性能数据。
