# SGN 配置与存储档位核心规范 修改记录报告

> **原始文件**: sgn_config_guide.md (v2.0)  
> **修改后文件**: sgn_config_guide_v2.1.md  
> **修改日期**: 2026-06-09  
> **修改性质**: 职责边界澄清与防误解补强（非逻辑修正，而是语义护栏）

---

## 一、修改背景

原始文档 v2.0 在 §4.1 和 §5 的表述中存在**语义混淆的隐患**：

- `stored_level` 字段被描述为「精度档位 0/2/4/6/8…」，容易让读者误以为它同时承担「跨档位截断/扩展决策依据」的职责。
- 单条 13 字节记录包含 `mode` + `stored_level`，容易让读者误以为它具有「自描述能力」，可以独立解析有效小数层数。

实际上：
- `stored_level` 存储的是**记录级 level 值**（合体语义中的整数倍数），在标准档位下恰好等于档位值，但本质上是 level 值。
- 跨档位兼容的**唯一决策依据**是文件头中的 `GLOBAL_LEVEL` 与当前编译的 `SGN_HPDC_LEVEL` 的比较。
- 单条记录**不自描述**，解析 `frac` 有效层数必须依赖文件头的 `GLOBAL_LEVEL`。

## 二、修改点清单

### 修改点 1：§4.1 单条合体记录 —— 增加职责澄清

**位置**: §4.1 表格后、「补零规则」段落处

**具体变更**:

1. **stored_level 字段描述修正**:
   - 原文: `uint32_t 小端，精度档位 0/2/4/6/8...`
   - 改文: `uint32_t 小端，本记录的 level 值（在标准档位文件中通常等于 0/2/4/6/8…）`

2. **补零规则主语修正**:
   - 原文: 以 `stored_level=2/4/6/0` 作为条件
   - 改文: 以 `GLOBAL_LEVEL=2/4/6/0` 作为条件，并注明「由文件头的 GLOBAL_LEVEL 决定，见 §4.2」

3. **新增 ⚠️ 职责澄清 框**:
   - 明确 `stored_level` 存储的是 level 值（256 倍数）
   - 明确跨档位决策依据是 `GLOBAL_LEVEL`，不是 `stored_level`
   - 明确单条记录不自描述，解析需先读文件头

### 修改点 2：§4.2 文件头 —— 强化 GLOBAL_LEVEL 唯一决策地位

**位置**: §4.2「自描述语义」列表后

**新增内容**:

> **注意**：`stored_level` 字段（见 §4.1）不参与跨档位兼容决策。所有截断/扩展行为仅由文件头的 `GLOBAL_LEVEL` 与当前编译的 `SGN_HPDC_LEVEL` 决定。

### 修改点 3：§5 跨档位兼容协议 —— 增加脚注

**位置**: §5 标题下

**新增内容**:

> **脚注**：本节中的「文件 level」均指文件头（§4.2）中的 `GLOBAL_LEVEL`，与单条记录内的 `stored_level` 无关。

## 四、被切断的两种典型误解

### 误解 1：stored_level 作为截断/扩展决策依据

**错误归纳**: 读者看到 §4.1 说 stored_level 是「精度档位」，又看到 §5 示例用 `stored_level=4`，会误以为单条记录自带档位信息，可以自决策。

**切断方式**: 
- §4.1 明确说明 stored_level 是「本记录的 level 值」，只是「在标准档位文件中通常等于档位值」
- §4.1 职责澄清第二条明确声明决策依据是 GLOBAL_LEVEL
- §4.2 新增「注意」段落再次强化

### 误解 2：单条记录具有自描述能力

**错误归纳**: 13 字节记录包含 mode + stored_level，看起来足够自描述。

**切断方式**:
- §4.1 职责澄清第三条明确声明「单条记录不提供完整的自描述信息」
- 强调解析 frac 有效层数必须先读文件头 GLOBAL_LEVEL

### §4.1 补零规则

**修改前**:
```markdown
**补零规则**:
- `stored_level=2`（FAST）: `frac[2..5]` 全部写 `0x00`
- `stored_level=4`（STD）: `frac[4..5]` 全部写 `0x00`
- `stored_level=6`（PREC）: `frac[0..5]` 全部有效
- `stored_level=0`（ARCHIVE）: `frac[0..5]` 全部写 `0x00`
```

**修改后**:
```markdown
**补零规则**（由文件头的 `GLOBAL_LEVEL` 决定，见 §4.2）：
- `GLOBAL_LEVEL=2`（FAST）: `frac[2..5]` 全部写 `0x00`
- `GLOBAL_LEVEL=4`（STD）: `frac[4..5]` 全部写 `0x00`
- `GLOBAL_LEVEL=6`（PREC）: `frac[0..5]` 全部有效
- `GLOBAL_LEVEL=0`（ARCHIVE）: `frac[0..5]` 全部写 `0x00`

> **⚠️ 职责澄清**
> 
> - `stored_level` 字段存储该记录的 `level` 值（即整数部分的 256 倍数），在标准档位文件中其值通常等于全局精度档位（0/2/4/6…）。  
> - **跨档位截断或扩展的决策依据是文件头中的 `GLOBAL_LEVEL` 与当前编译精度 `SGN_HPDC_LEVEL` 的比较，不是本记录的 `stored_level`。**  
> - 单条记录不提供完整的自描述信息。要正确解析 `frac` 的有效层数，必须先读取文件头的 `GLOBAL_LEVEL`。
```

## 六、结论

本次修改属于**文档严谨性补强**，未改变任何技术逻辑或 ABI 契约。通过增加职责边界说明，切断了读者可能产生的两类错误归纳：

1. `stored_level` ≠ 跨档位决策依据（唯一决策依据是 `GLOBAL_LEVEL`）
2. 单条记录 ≠ 自描述（解析需依赖文件头上下文）

修改后文档更符合「唯一权威源」的自我要求，降低了扩展开发者或维护者因「望文生义」而引入实现错误的风险。

## 附录 A：数学分析报告勘误（2026-06-09 补充）

**原报告**: `sgn_math_analysis_report.md` §3.4  
**错误内容**: 
> 补零扩展后，两个点的距离可能从 256^{-5} 变为 256^{-7}（如果差异在新增层）

**错误性质**: 事实性错误。补零扩展的新增层全部写 0，对任意两点都相同，不可能引入新的首次差异层。因此补零扩展不会改变超度量距离。

**正确表述**:
> 补零扩展是严格的等距嵌入（isometry）。对任意 `X, Y ∈ HC(p)`，设首次差异层为 `k`，补零扩展到 `HC(q)`（`p < q`）后，首次差异层仍为 `k`，距离保持 `256^{-k}` 不变。只有截断降级（`p > q`）可能丢失深层差异，使距离从 `256^{-k}` 坍缩为 `0`（当 `k ≥ q` 时）。

**修正文件**: `sgn_math_analysis_report_v2.md`
