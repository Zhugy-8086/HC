# SGN 配置规范数学严谨性分析报告

## 一、当前修改的充分性评估

### 1.1 修改已解决的问题
- 切断了「stored_level 参与跨档位决策」的误解路径 ✓
- 切断了「单条记录自描述」的误解路径 ✓
- 明确了 GLOBAL_LEVEL 的唯一决策地位 ✓

### 1.2 修改未触及的深层数学问题

**问题 A：语义重载（Semantic Overloading）**

`stored_level` 字段在数学上同时承载两个类型：
- Type L: level ∈ ℕ（无界自然数，合体语义）
- Type P: precision ∈ {0, 2, 4, 6, 8}（精度档位，文件格式语义）

当前文档说「在标准档位文件中通常等于 0/2/4/6/8」，这仍然是**偶然同构**（accidental isomorphism）的暗示。数学上，一个字段同时属于两个类型空间 L 和 P，且 P ⊂ L（精度档位是 level 值的子集），这是类型系统的耦合。

**问题 B：§5 示例的数学歧义**

§5.1 示例：
```
文件: stored_level=4, frac=[a,b,c,d,0,0]
当前: SGN_HPDC_LEVEL=6
```

此处用 `stored_level=4` 指代文件档位，但 `stored_level` 在数学上应表示某条记录的 level 值。如果文件中存在一条记录的 `stored_level=1000`（level=1000），按照此示例的写法，读者会困惑：「这条记录的 stored_level=1000，是不是意味着精度档位 1000？」

**问题 C：投影算子的数学缺失**

跨档位兼容在数学上是：
- 补零扩展：等距嵌入 ι: HC_p → HC_q (p < q)
- 截断降级：投影算子 π: HC_p → HC_q (p > q)

文档使用工程语言（「补零」「截断」），但未定义这些操作的数学性质：
- 幂等性：π(π(x)) = π(x)
- 单调性：x ≤ y ⇒ π(x) ≤ π(y)
- 核空间：Ker(π) = {x | x 的低位非零，高位为零}

**问题 D：物理值公式的档位依赖性**

§1.1 给出物理值公式：
V = level × 256 + int_part + Σ frac[i] / 256^{i+1}

但该公式隐含假设了所有 6 层 frac 都有效。当精度档位 p=2 时，frac[2..5] 为零，公式仍然成立，但「有效层数」的数学定义未显式给出。应该定义为：

V_p = level × 256 + int_part + Σ_{i=0}^{p-1} frac[i] / 256^{i+1}  （p ∈ {0,2,4,6,8}）

其中 p=0 时求和项为空，退化为纯整数。

## 二、改进方案对比（从数学优雅性角度）

### 方案 0：当前修改（语义护栏）
- 优点：零 ABI 变更，风险最低
- 缺点：未解决语义重载的根本问题，「通常等于」的表述仍暗示偶然同构

### 方案 1：数学化重构（推荐）
在 §1.1 物理值公式处引入档位依赖的显式定义：

```
V(level, int_part, frac[], p) = level × 256 + int_part + Σ_{i=0}^{p-1} frac[i] / 256^{i+1}

其中 p ∈ {0, 2, 4, 6, 8} 为有效小数层数（精度档位）。
当 p < 6 时，frac[p..5] 在数学上属于核空间 Ker(π_p)，其值为 0。
```

在 §4.1 处彻底切断 stored_level 与档位的联系：

```
9       4     level           uint32_t 小端，本记录的 level 值（合体语义）
                              注意：此字段仅表示 level × 256 的倍数，
                              与文件精度档位无关。精度档位由文件头 GLOBAL_LEVEL 决定。
```

在 §5 处引入投影算子符号：

```
**数学定义**：
设 HC(p) 为 p 层精度空间。定义投影算子 π_{q←p}: HC(p) → HC(q)（p > q）：
  π_{q←p}(level, int, frac[0..p-1]) = (level, int, frac[0..q-1])

性质：
1. 幂等：π_{q←p} ∘ π_{p←r} = π_{q←r}  （q < p < r）
2. 单调：x ≤ y ⇒ π(x) ≤ π(y)  （保持超度量序）
3. 误差界：|x - π(x)| < 256^{-(q+1)}  （截断误差有界）
```

### 方案 2：字段更名（ABI 微变更）
将 `stored_level` 更名为 `level`，彻底消除「stored_level 暗示精度档位」的联想。

文件格式变更：
```
9       4     level           uint32_t 小端，level 值
```

此方案需要更新所有引用该字段的文档，但数学上最清晰。

### 方案 3：自描述增强（格式变更）
如果坚持单条记录自描述，应增加独立的 `precision` 字段（1 字节）：

```
Offset  Size  Field           Description
─────────────────────────────────────────────────
0       1     mode            0x02 = 合体模式
1       1     sign            0=正，1=负
2       1     int_part        整数部分 0~255
3       1     frac[0]         小数第 0 层
...     ...   ...             ...
9       4     level           uint32_t 小端，level 值
13      1     precision       有效小数层数 0/2/4/6/8
─────────────────────────────────────────────────
Total: 14 字节
```

代价：记录长度从 13 变为 14 字节，Flash 地址计算需调整。
收益：单条记录真正自描述，无需依赖文件头即可解析。

### 方案 4：编码复用（折中）
利用 `level` 字段的取值范围特性进行编码：

```
if level <= 8 且 level ∈ {0,2,4,6,8}:
    precision = level  （兼容标准档位）
else:
    precision = GLOBAL_LEVEL  （非标准 level 值，依赖文件头）
```

此方案在数学上定义了一个分段函数，保持了向后兼容，但增加了解析复杂度。

## 三、SGN 全栈数学隐患扫描

### 3.1 sgn_math_core.md：超度量距离的完备性

**隐患**：§2.2 强三角不等式证明假设了所有元组长度相同（L 固定）。但在合体模式下，当 level 进位导致「无限深度树」时，两个不同 level 的数比较时：

```
A = (1, 0, 0, ...)
B = (0, 255, 255, ...)
```

按字典序：level_A=1 > level_B=0，所以 A > B。
但物理值：V(A) = 256，V(B) = 255.999...，A > B 仍然成立。

**问题**：当 int_part 溢出向 level 进位时，超度量距离是否仍然保持？
- 仅 HC 模式：是，有限深度树，完备
- 合体模式：进位导致「逃逸」，超度量球撕裂，距离定义失效

**建议**：在 sgn_math_core.md 中增加「合体模式非保距」的显式定理。

### 3.2 sgn_hc_ops.md：饱和加法的数学性质

**隐患**：§3.1 饱和加法定义为 `int_part clamp 255`。数学上：
```
add_sat(a, b) = min(a + b, 255.999...)
```

这**不是**阿贝尔群运算，因为：
- 无逆元：若 a=200, b=100，add_sat(a,b)=255.999，但不存在 c 使得 add_sat(255.999, c)=200
- 结合律仍成立：add_sat(add_sat(a,b),c) = add_sat(a, add_sat(b,c))

**建议**：明确声明饱和加法构成**半环**（semiring），而非群。这在数学上影响软阈值衰减的收敛性分析。

### 3.3 sgn_config_guide.md：档位表的数学间隙

**隐患**：档位表定义了 FAST=2, STD=4, PREC=6, FUTURE=8，但：
- 档位的步长是 2（层数），不是 1
- 为什么不是 1, 3, 5, 7？数学上 2 的倍数对应 16 位精度步长（2 层 = 16 位）

**建议**：在档位表中增加「每层 8 位，每档 2 层 = 16 位精度」的数学说明，使档位值的数学基础显式化。

### 3.4 跨档位兼容的隐藏假设

**隐患**：§5.1 补零扩展声称「物理值不变」。数学上：
```
V_4 = level×256 + int + a/256 + b/256² + c/256³ + d/256⁴
V_6 = level×256 + int + a/256 + b/256² + c/256³ + d/256⁴ + 0/256⁵ + 0/256⁶
```

确实 V_4 = V_6，但**超度量距离**呢？

**补零扩展（p < q）**：对任意 `X, Y ∈ HC(p)`，设首次差异层为 `k`（`0 ≤ k ≤ p-1`）。补零扩展到 `HC(q)` 后，前 `p` 层保持不变，新增层 `p..q-1` 全为 0（对 `X, Y` 相同），因此首次差异层仍为 `k`，距离不变：
```
d_{HC(q)}(ι(X), ι(Y)) = 256^{-k} = d_{HC(p)}(X, Y)
```
**结论**：补零扩展是严格的**等距嵌入**（isometry），不会改变超度量距离。

**截断降级（p > q）**：仅当首次差异层 `k ≥ q` 时，前 `q` 层完全相同，截断后 `π(X) = π(Y)`，距离从 `256^{-k}` 坍缩为 `0`；若 `k < q`，距离保持 `256^{-k}` 不变。因此截断降级是**非单射**的，可能丢失深层差异，但不会使距离「变大」。

**建议**：在 §5 中明确此性质。

### 3.5 文件头 GLOBAL_LEVEL 的集合论问题

**隐患**：文件头声称「全局精度档位」，但文件中的 N 条记录是否允许有不同的「实际精度」？

如果所有记录必须服从 GLOBAL_LEVEL，那么 stored_level（level 值）与 GLOBAL_LEVEL 无关，但 frac 的补零规则统一由 GLOBAL_LEVEL 决定。这在数学上是：
```
文件 F = (GLOBAL_LEVEL=p, {record_i})
record_i = (mode, sign, int, frac[0..5], level_i)
其中 frac_i[j] = 0 for all j >= p
```

这是一个**约束系统**，但文档未明确说明这是硬性约束还是软性约定。如果某条记录的 frac[5] 非零而 GLOBAL_LEVEL=4，是错误还是未定义行为？

**建议**：在 §4.2 增加「文件一致性约束」：所有记录的 frac[GLOBAL_LEVEL..5] 必须为 0，加载时可选择验证或忽略。

## 四、综合建议与最优路径

### 4.1 对当前修改的最终评价

当前 v2.1 修改（语义护栏）在**工程层面**是充分且必要的，但在**数学层面**属于「症状缓解」而非「病因根除」。

根本病因：`stored_level` 字段的命名和类型定义在数学上是不良的（ill-typed）。

### 4.2 推荐的最小修改方案（v2.1 补丁）

在不改变 ABI 的前提下，通过纯文档层面的数学化重构，消除歧义：

**步骤 1：§4.1 字段更名（文档层面）**
将表格中的 `stored_level` 在文档中统一称为 `level` 或 `level_value`，仅在括号中注明「物理存储字段名为 stored_level，历史原因」。

**步骤 2：§5 示例修正**
将示例中的 `stored_level=4` 改为 `GLOBAL_LEVEL=4, 某记录 level=0`（或 `level=4` 如果该记录的 level 恰好为 4）。明确区分文件级变量和记录级变量。

**步骤 3：增加投影算子数学定义（§5 开头）**

```markdown
### 5.0 数学预备：投影与嵌入

设 HC(p) 为 p 层有效小数精度空间（p ∈ {0,2,4,6,8}）。

**嵌入算子** ι_{q←p}: HC(p) → HC(q) （p < q）：
  ι(x) = x  with  frac[p..q-1] := 0

**投影算子** π_{q←p}: HC(p) → HC(q) （p > q）：
  π(x) = x  with  frac[q..p-1] discarded

**定理**（等距嵌入）：对任意 x, y ∈ HC(p)，嵌入到 HC(q)（q > p）后：
  d_{HC(p)}(x, y) = d_{HC(q)}(ι(x), ι(y))  当 k(x,y) ≤ p+1

即：在原有精度层内，超度量距离保持不变。新增层仅对更精细的差异敏感。
```

### 4.3 推荐的根本修改方案（v3.0 远期）

如果未来允许 ABI 变更，建议：

1. **字段更名**：`stored_level` → `level`（uint32_t），语义唯一化
2. **文件头强化**：`GLOBAL_LEVEL` → `precision`（uint8_t），明确为精度档位
3. **增加一致性校验**：加载时验证 ∀i, frac_i[GLOBAL_LEVEL..5] == 0

## 五、结论

1. **当前修改合理**：语义护栏切断了读者误解，是工程文档的正确做法。
2. **数学层面仍有隐患**：`stored_level` 的语义重载、投影算子的未定义、文件一致性约束的缺失，是更深层的数学问题。
3. **最优路径**：当前采用 v2.1 语义护栏（零风险），远期规划 v3.0 数学重构（字段更名 + 算子定义 + 一致性约束）。
4. **核心原则**：SGN 的价值在于「纯整数、确定性、超度量」。任何文档层面的模糊都会破坏「确定性」，因为确定性不仅是实现属性，也是**规范属性**——规范必须让读者产生唯一确定的数学理解。
