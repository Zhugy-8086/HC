SGN ABI 规范编制计划书

版本: 1.0 (草案)
日期: 2026-06-09
状态: 规划阶段，尚未编写正式 ABI 文件
目的: 明确 SGN 技术栈 ABI 规范的范围、内容、稳定性承诺及扩展机制

---

1. 背景与目标

SGN 技术栈已有 9 份核心规范文档（v2.1 数学澄清版）和 14 份扩展文档。这些文档定义了数据结构、算法、存储格式、网络协议等，但尚未以统一的 ABI（Application Binary Interface）规范形式固化。

ABI 规范的目标是：

· 为所有实现（C++、Python、Rust、MCU 汇编等）提供二进制级别的接口契约。
· 确保不同编译器、不同平台、不同语言编写的模块能够互操作（例如 PC 端生成的模型文件可在 MCU 端直接加载）。
· 明确哪些接口是稳定版本，哪些是实验性扩展，避免生态分裂。
· 为未来插件机制（软编码、动态加载）预留扩展点。

本次计划书不输出具体 ABI 文件，仅规划其内容、结构、编写步骤和扩展策略。

---

2. 范围界定

ABI 规范仅覆盖以下 9 个核心文件中定义的、需要进行跨平台/跨语言交互的接口：

文件 ABI 覆盖内容
sgn_config_guide.md 模型文件二进制布局（13 字节记录、23 字节文件头）、精度档位枚举、内存互斥宏
sgn_math_core.md 无直接 ABI（纯数学定义，不产生函数接口）
sgn_hc_ops.md HC8/HC16 比较、加法、减法、软阈值等基础运算的函数签名
sgn_trie_core.md Trie 节点结构、查找/插入/遍历接口
sgn_engine_core.md WTA、软阈值衰减、LRU、形态学等算法接口
sgn_storage_core.md RLE 编解码、Merkle 树更新、RS 纠错、校验和计算接口
sgn_network_core.md UART 帧打包/解析、Lamport 时钟、Quorum 共识、看门狗接口
sgn_reliability_core.md TMR 投票、校验和增量更新接口
sgn_index_matrix.md 无直接 ABI（索引文档）

不覆盖：14 份扩展规范（如平方根、补码、投影除法等），因为它们属于“软编码/插件”范畴，不要求所有实现必须提供。

3. ABI 规范的内容结构

每个 ABI 接口定义应包含以下信息（以 C/C++ 头文件风格描述，但不限定语言）：

3.1 基础类型定义

· 定长整数类型（uint8_t, uint16_t, uint32_t, uint64_t）及其字节序（小端）。
· HC 相关结构体：hc8_t, hc16_t, leveled_hpdc8, shc8_t 的内存布局（字段顺序、对齐要求）。
· Trie 节点结构 HCTrieNode 的精确内存布局（包括 union 的选择策略，由编译期宏控制）。
· 枚举常量：sgn_scan_mode_t, sgn_verify_result_t 等。

3.2 函数接口（仅列出必须导出的符号）

每个函数应声明：

· 函数名（修饰后名称，如 C++ extern "C" 或平坦命名空间）。
· 参数类型、顺序、传递方式（值/指针）。
· 返回值类型。
· 所属模块（如 sgn_hc_ops, sgn_storage_core）。
· 是否为纯函数（无副作用）、是否可重入、线程安全要求。

示例（伪代码）：

```c
// sgn_hc_ops.h
bool sgn_hc8_less(const uint8_t a[6], const uint8_t b[6]);
void sgn_hc8_add_saturated(const uint8_t a[6], const uint8_t b[6], uint8_t out[6]);
```

3.3 内存布局与对齐

· 所有结构体按 1 字节对齐（无填充），便于跨平台。
· 数组长度固定（如 hc8_t 固定 6 字节），不使用变长数组或指针。

3.4 错误处理与未定义行为

· 明确哪些输入会导致未定义行为（如 sgn_hc8_sub 要求 a>=b）。
· 定义错误码枚举（如 SGN_OK, SGN_ERR_OUT_OF_RANGE, SGN_ERR_TMR_MULTI_FAULT），但仅用于可选校验接口。

3.5 版本管理宏

· SGN_ABI_VERSION_MAJOR, SGN_ABI_VERSION_MINOR, SGN_ABI_VERSION_PATCH。
· 运行时版本检查函数（如 uint32_t sgn_abi_version(void)）。

---

4. 稳定性承诺

ABI 规范采用语义化版本控制：

· 主版本号（Major）：不兼容的 ABI 变更（如结构体字段重排、函数删除）。
· 次版本号（Minor）：新增接口，向后兼容（原有接口不变）。
· 修订号（Patch）：仅文档修正、注释澄清，不改变二进制布局。

冻结期策略：一旦发布 v1.0，未来 3 年内不再增加主版本号。若需增加新功能，以“扩展插件”形式提供，不修改核心 ABI。

---

5. 扩展机制（软编码 / 插件）

为了避免核心 ABI 无限膨胀，所有非必需功能均以“软编码”或“插件”形式实现：

· 软编码：扩展功能不纳入 ABI，而是由上层应用通过调用核心 ABI 的基础接口自行组合实现（例如投影除法、平方根等）。
· 插件：允许动态加载共享库（.so, .dll），通过统一的 sgn_plugin_t 接口注册扩展函数。插件不要求跨平台二进制兼容，由同一编译器/同一平台生成即可。

ABI 需要提供的插件支持：

· 插件注册表（函数指针数组）。
· 版本匹配检查。
· 资源清理回调。

插件功能示例：

· 自定义扫描模式（如螺旋扫描）。
· 自定义模板合并策略（非 OR/AND）。
· 自定义校验算法（非交叉相乘）。

---

6. 多语言绑定策略

ABI 规范的核心是 C 语言接口（因为几乎所有语言都能调用 C ABI）。具体做法：

· 所有结构体使用 typedef struct，不包含成员函数。
· 所有函数使用 extern "C" 修饰，避免 C++ 名称修饰。
· 提供纯 C 头文件（.h），不依赖 C++ 标准库。

对于其他语言（Python、Rust、Go）的绑定，由社区提供独立的“包装库”，不纳入 ABI 规范。

7. 编写步骤与交付物

阶段 1：抽取 ABI 清单（1 天）

· 从 9 个核心文件中列出所有需要跨平台暴露的类型、常量、函数。
· 标注哪些是必须的，哪些是可选的（如调试用接口）。

阶段 2：定义二进制布局（1 天）

· 精确描述每个结构体的字节偏移和大小。
· 定义枚举值（如 sgn_scan_mode_t = {0,1,2}）。

阶段 3：编写 ABI 头文件草案（2 天）

· 产出 sgn_abi.h（核心类型和函数声明）。
· 产出 sgn_abi_plugin.h（插件注册接口）。

阶段 4：编写 ABI 规范文档（1 天）

· 将上述内容以纯文本形式写成 sgn_abi_spec.md，包含：
  · 版本号。
  · 类型和函数的文字描述。
  · 内存布局图示。
  · 版本兼容性承诺。

阶段 5：评审与冻结（1 天）

· 内部评审，确保所有实现（Python 版、C++ 主引擎、MCU 固件）对齐。
· 发布 v1.0 草案，冻结 1 个月供社区反馈。

---

8. 与现有规范的关系

· sgn_config_guide.md 中的二进制文件格式直接成为 ABI 的一部分。
· sgn_hc_ops.md 中的函数签名直接映射为 ABI 函数。
· sgn_engine_core.md 中的算法接口（如 WTA）也纳入 ABI，但允许实现内部优化（只要外部调用行为一致）。
· 扩展规范（如平方根、补码等）不进入 ABI，仅作为文档参考。

---

9. 风险与应对

风险 应对措施
不同编译器对结构体对齐处理不同 显式使用 #pragma pack(1) 或 __attribute__((packed))
未来需要修改核心数据结构 通过增加新版本结构体（如 hc8_v2_t）并弃用旧版，不原地修改
插件机制导致生态碎片化 插件仅用于实验功能，核心 ABI 保持精简；插件不保证跨平台兼容
部分实现（如 Python 版）性能低于 C Python 版可调用 C 写的 ABI 共享库，不要求原生实现
