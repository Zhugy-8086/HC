1. 概述

当前 SGN 代码仓库（以 hpdc_core.cpp、hpdc_sandbox.cpp、hpdc_cpp.hpp 等为代表）在工程实现层面存在显著的“空心化”现象。尽管设计文档（sgn_math_core.md、sgn_hc_ops.md、sgn_engine_core.md 等）已完整定义了 HC8/16/32/64 的数学性质、运算规则及引擎算法，但 C 层实现仅完整覆盖了 HC8 一个变体。与此同时，顶层 C++ 包装器与沙箱模块宣称支持全部类型，实则因底层缺失而依赖临时内联实现或直接被静态断言禁用。

本文件目的：

· 系统性地罗列所有“已声明但未实现”的 C 层函数接口。
· 指出现有“大杂烩”式文件组织（单文件包含所有类型）对后续开发、维护及嵌入式部署造成的结构性阻碍。
· 提出“按 HC 类型拆分模块”（hc8.c、hc16.c、hc32.c、hc64.c）的具体架构蓝图。
· 阐明 C++ 统一包装层（hpdc_cpp.hpp）与沙箱（hpdc_sandbox.cpp）因底层缺失而存在的“空心化”问题。
· 给出可选的指令集（SIMD）优化路径与其实施前提。
· 制定分阶段实施路线图，确保工程演进有序、可回溯。

本文件不重复 问题.md 中的任何错误修正项。读者应将两份文件视为同一问题的两个正交维度：正确性与完备性。

2. 当前工程结构的根本缺陷

2.1 一体化 C 文件导致链接膨胀与编译耦合

当前所有 HC 变体的核心运算全部堆砌于 hpdc_core.cpp 单个编译单元中。其直接后果包括：

· 链接膨胀：即使用户仅需 HC16 进行时间戳运算，也必须将包含 HC8/32/64/DC/SHC8/Leveled 全部符号的整个目标文件链接进最终镜像。在资源受限的嵌入式平台（如 AI8051U，Flash 64KB/SRAM 34KB）上，这种“全量包含”模式直接挤占宝贵的存储空间。
· 编译耦合：修改 HC32 的 to_double 转换精度将触发整个 hpdc_core.cpp 的重编译。在大型项目中，这种耦合显著拉长增量编译时间。
· 符号命名污染：所有类型共处同一文件，容易因宏定义或静态函数作用域管控不严产生意外的符号冲突。

2.2 补丁层（sgn_abi_dynamic.cpp）与核心层重复定义

当前仓库中同时存在 hpdc_core.cpp 与 sgn_abi_dynamic.cpp 两个文件，且两者均定义了 sgn_hc8_to_double、sgn_hc16_to_double 等基础转换函数。这种设计原本意图作为“过渡补丁”以修复 HC16 物理值 bug，但实际导致了严重的链接二义性：

· 若同时链接两个目标文件，链接器可能报出“多重定义”错误（具体取决于链接器行为与 weak 属性设置）。
· 若仅链接其一，用户可能错失 HC16 的 bug 修复（sgn_abi_dynamic.cpp 中的通用 sgn_hc_physical_value 修复了 scale 起点问题，而 hpdc_core.cpp 中的 sgn_hc16_to_double 仍为旧版本）。
· 这种“两份实现”的状态使 ABI 稳定性无法得到保证。

2.3 C++ 统一包装层（hpdc_cpp.hpp）与底层 C 实现脱节

hpdc_cpp.hpp 是本项目官方推荐的 C++ RAII 入口，其通过 HC<RawT> 模板与 hc_api_traits 特化试图为所有 HC 类型提供统一接口。然而，由于底层 C 层缺失函数实现，该包装层呈现“空心化”特征：

· HC8：hc_api_traits<sgn_hc8_t> 完整映射至 sgn_hc8_add_saturated 等 C 函数，工作正常。
· HC16：C 层仅有 sgn_hc16_less、sgn_hc16_equal 及 sgn_hc16_to_double。包装器中的 add_sat、sub、soft_threshold 被迫在模板层以临时内联算法（uint32_t carry 循环）补足。这导致了 “双轨制”：HC16 的运算逻辑存在 C 层（无实现）与 C++ 层（内联实现）两套定义，一旦未来 C 层补全函数，接口签名若不一致将引发 ABI 断裂。
· HC32 / HC64：C 层完全不提供任何运算函数。包装器中的 hc_api_traits 仅提供 to_double/from_double（通用推导）及 zero/max，而 add_sat、less 等运算符通过 static_assert 被显式禁用。这意味着用户无法在 C++ 中对 HC32/HC64 执行加法或比较，严重限制了高级数据结构的泛型编程。

2.4 沙箱（Sandbox）仅完整支持 HC8，且实现自包含

投影沙盒（hpdc_sandbox.cpp）是 HPDC 体系的“浮点编程魔法”入口，但其现状与 C++ 包装层类似：

· 投影/逆投影函数（sgn_sandbox_project_hc8 / inverse_hc8）因依赖 HC8 完整实现而表现正常。
· HC16/HC32 的沙盒函数虽已声明，但其内部物理值计算（hc16_physical_value、hc32_physical_value）是在沙盒模块内部独立重新实现的静态函数，并未复用 hpdc_core.cpp 或 sgn_abi_dynamic.cpp 中的官方转换接口。这直接导致了代码冗余与维护风险——若核心层的转换算法修正（如 HC16 scale bug），沙盒内部的重复代码将不会被同步修正。
· 更为严重的是 int_bits 参数与 HC 类型位宽之间的无联动校验。sgn_sandbox_create(prec, int_bits) 允许用户随意指定 int_bits（如 HC8 传入 16），此时 R = 2^16 = 65536，而 HC8 最大物理值仅为 256。逆投影时几乎所有浮点数都被饱和至 HC8_MAX，但沙盒既不报错也不警告，属于“静默失效”。

3. 目标架构：按 HC 类型模块化拆分

3.1 C 层目录与文件规划

为解决 §2.1 的耦合问题，建议将现有单一体文件按 HC 类型（即元数据表中的 kind）拆分为独立编译单元。目录结构规划如下：

```
sgn/
├── include/
│   ├── sgn/
│   │   ├── hc.h               # 通用基础设施：sgn_hc_meta_t 元数据查询、错误码、宏定义
│   │   ├── hc8.h              # HC8 类型声明、运算接口（比较/加/减/软阈值/转换）
│   │   ├── hc16.h             # HC16 类型声明与接口
│   │   ├── hc32.h             # HC32 类型声明与接口
│   │   ├── hc64.h             # HC64 类型声明与接口
│   │   ├── shc.h              # 有符号扩展（补码 SHC8）[可选]
│   │   ├── dc.h               # 十进制定点数 DC [可选]
│   │   └── leveled.h          # 合体模式（level + HC）[可选]
│   └── sgn.h                  # 主入口：包含所有头文件（仅 PC 开发/测试用）
├── src/
│   ├── hc.c                   # 通用辅助（CRC、GF256、元数据查表）
│   ├── hc8.c                  # HC8 所有运算实现
│   ├── hc16.c                 # HC16 所有运算实现
│   ├── hc32.c                 # HC32 所有运算实现
│   ├── hc64.c                 # HC64 所有运算实现
│   ├── shc.c                  # 有符号运算 [可选]
│   ├── dc.c                   # DC 运算 [可选]
│   └── leveled.c              # 合体运算 [可选]
└── tests/
    ├── test_hc8.c
    ├── test_hc16.c
    └── ...
```

核心优势：

1. 按需链接：嵌入式项目若仅使用 HC16，只需编译并链接 hc16.c + hc.c（约 2KB），不再被 HC8/32/64 的符号拖累。
2. 独立演进：修复 HC32 的转换精度不会触发 HC8 的重编译，多人协作时冲突域大幅缩小。
3. 并行验证：各类型的单元测试可独立运行，便于定位回归缺陷。

3.2 C++ 层配套调整

hpdc_cpp.hpp 不应改变其“单一入口”的用户体验，但内部实现必须从当前“内联补齐”切换为“薄包装”：

· hc_api_traits<RawT> 中的 add_sat、sub、less 等函数应直接调用 hcX.h 中声明的 C 函数（如 sgn_hc16_add_saturated），不再保留模板内联算法。
· 对于尚未实现的 C 函数（如 HC64 全部运算），应提供明确的 #error 或 static_assert，而不是静默提供有缺陷的通用回退，以此暴露依赖关系，方便开发者按模块补齐。

3.3 沙箱（Sandbox）依赖反转

沙箱模块必须从“自包含实现”转为“依赖抽象接口”：

· 删除沙箱内部的 hc8_physical_value、hc16_physical_value 等静态重复实现。
· 统一调用 hcX.h 中导出的 sgn_hcX_to_double 与 sgn_double_to_hcX。
· sgn_sandbox_create 必须增加对 int_bits 的合法性校验：
  · HC8 要求 int_bits ≤ 8；HC16 要求 ≤ 16；HC32 要求 ≤ 32；HC64 要求 ≤ 64。
  · 若传入参数越界，应返回 NULL 或设置错误码，不得静默降级。
· 同时提供 sgn_sandbox_create_auto(sgn_hc_kind_t kind) 便捷构造，自动调用 sgn_sandbox_default_int_bits(kind) 填入合适的位宽。

难度分析：补全上述所有缺失函数属于 “机械式代码生成” 范畴。所有算法模式（从最低层起算的进位/借位循环、字典序比较、软阈值饱和逻辑）在 sgn_hc_ops.md 中已有完整伪代码。开发者仅需将 HC8 实现中的 uint8_t 替换为 uint16_t / uint32_t / uint64_t，将层数 6 替换为 4 / 3 / 2，并将基数 256 替换为 65536 / 2^32 / 2^64。该工作不涉及数学推导或新算法设计，纯属工程实现体力活。

5. 规范文档（.md）与 C 代码的工程脱节

设计规范（sgn_engine_core.md、sgn_storage_core.md、sgn_network_core.md）已完成大量高级算法定义，但对应的 C 实现文件（hpdc_engine.cpp、hpdc_storage.cpp、hpdc_network.cpp）中存在大面积占位符或虚假模拟数据：

· 引擎层（hpdc_engine.cpp）：
  · sgn_wta_compete 中的相似度 sim 直接使用 (candidates[i] & 0x3F) 模拟，并非真正的 Hamming 距离计算。
  · sgn_dilate / erode / open / close 所有形态学函数仅将 result_count 置 0，无任何实质性遍历或标记逻辑。
· 存储层（hpdc_storage.cpp）：
  · sgn_rle_encode / decode 仅返回入参长度或 0，未实现压缩/解压核心循环。
  · sgn_file_write / read 为空壳（(void) 参数，返回 0）。
· 网络层（hpdc_network.cpp）：
  · 虽然 UART 帧打包/解析和 COBS 编解码已有具体实现，但 Lamport 时钟进位处理不完整（ss[0] 溢出未向 ss[1] 进位，导致时钟很快失去单调性）。

上述脱节意味着 设计蓝图（规范文档）与交付物（C 代码）之间存在巨大鸿沟。在完成 §3 的文件拆分与 §4 的标量运算补齐后，应按模块逐一将规范文档中的伪代码翻译为 C 实现，并移除所有占位模拟逻辑。

6. 指令集优化（SIMD）的可行路径

当前所有 HC 运算均采用标量循环（Scalar Loop），未使用任何 SIMD 指令集（SSE/AVX/NEON）。从体系结构角度，HC 类型天然具备 SIMD 友好性：

· HC16 总长 8 字节，两个 HC16 实例恰好填满一个 128 位 SSE 寄存器（__m128i），可同时执行两组 16 位通道的加法。
· HC64 总长 16 字节，单个 HC64 实例即可独占一个 SSE 寄存器，无需拆包即可进行 64 位整数向量运算。

主要障碍（横向进位）：HC 加法中最耗时的部分是将最低层（frac[L-1]）的进位逐层向上传播。SIMD 擅长“纵向并行”（多组数据的同层相加），而不擅长“横向依赖”（同组数据内的层间进位）。为实现高效 SIMD，需将内存布局从当前的 AoS（Array of Structures，结构体数组） 切换为 SoA（Structure of Arrays，数组结构体）：

· AoS：struct { uint16_t v0, v1, v2, v3; } arr[N];
· SoA：uint16_t v0[N], v1[N], v2[N], v3[N];

在 SoA 布局下，一次 _mm_add_epi64 可同时处理 N 个 HC16 的同一层，进位掩码可通过向量比较（_mm_cmpgt_epi64）批量生成，再通过移位指令（_mm_srli_si128）将低层进位向量“搬移”到高层，最后用 _mm_add_epi64 完成进位叠加。

实施建议：SIMD 优化属于“性能增强”而非“功能必需”。应在标量实现全部补齐、单元测试全部通过后，以独立补丁形式引入。初期不建议同时维护 AoS 与 SoA 两套代码，可选择编译期宏（#ifdef SGN_USE_SIMD）进行切换，但需明确 SoA 会改变内存布局，与现有文件 I/O 及 C++ 包装器可能不兼容。

7. 补丁层（sgn_abi_dynamic）的最终处理

问题.md 已指出 sgn_abi_dynamic 与核心层的冲突。在本重构蓝图中，该补丁层的处理方式如下：

1. 剥离元数据：将 sgn_hc_meta_t 元数据表及 sgn_hc_meta_get 函数移入新的 hc.c 通用模块，作为所有 HC 类型的基础配置源。
2. 合并修复：将 sgn_abi_dynamic.cpp 中经修正的 sgn_hc_physical_value（通用算法，修复 HC16 scale bug）与 sgn_double_to_hc（消除硬编码上限）分别下沉至 hcX.c 中的 to_double 与 from_double 实现。
3. 删除补丁文件：完成上述迁移后，从仓库中彻底移除 sgn_abi_dynamic.h 与 sgn_abi_dynamic.cpp，杜绝后续链接冲突风险。

9. 与问题1（问题.md）的边界界定

为确保开发人员不会混淆两类工作，特此明确区分：

维度 问题1（问题.md） 问题2（本文件）
性质 正确性缺陷（Bug） 完备性缺失 + 结构优化
典型条目 Trie 内存池越界写入、RS 解码错误值计算、Lamport ss[0] 无进位 HC16 没有 add_sat 函数、沙箱 int_bits 无校验、文件未拆分、无 SIMD
修复优先级 最高（阻断功能） 中高（影响开发效率与部署灵活性）
执行顺序 必须先修复，否则后续重构无意义 建议在问题1修复完成后立即启动
